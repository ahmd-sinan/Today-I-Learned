# Tries: Trees of Arrays 

**Date:** 2026-04-05

Today I learned about **Tries**, which are another form of data structure. If Hash Tables are arrays of linked lists, Tries are essentially **trees of arrays**.

## 1. The Ultimate Speed: $O(1)$ 
- The biggest advantage of a Trie is its speed. Tries are always searchable in constant time, $O(1)$. 

- Even if you have a dictionary of 10 million words, searching for the word "Toad" only takes exactly 4 steps (one for each letter). The search time is determined by the length of the word, not the amount of data stored! This structure offers a search time of $O(1)$.

## 2. The Fatal Flaw: Massive Memory Cost 
- The downside of this structure is how many resources are required to use it. One major downside to Tries is that they tend to take up a large amount of memory.

- Because every single letter node must contain an array of pointers to every *possible* next letter (usually 26 for the English alphabet), the memory footprint explodes quickly. 
    * Notice that we need $26 \times 4 = 104$ nodes just to store the word "Toad"!.

## 3. How Paths are Stored and Shared 
- When "Toad" is stored, it creates an array for the first letter, points 'T' to a second array, points 'O' to a third, points 'A' to a fourth, and marks 'D' as the end.

- The magic happens when words overlap. If we add the word "Tom" after "Toad", we don't need to create new arrays for 'T' and 'O'. "Tom" simply follows the existing path for 'T' and 'O', and then branches off at the third array to create a new path for 'M'.
